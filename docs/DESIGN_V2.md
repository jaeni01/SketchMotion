## 1. 시스템 개요

```
                        ┌───────────────────────────────────────┐
                        │  PC — SketchMotion HMI (MFC, C++20)   │
                        │  ┌─────────┐ ┌─────────┐ ┌─────────┐  │
   AR0234 90fps ─USB──▶ │  │ Vision  │ │ Canvas  │ │ Motion  │  │
   (글로벌 셔터,        │  │ Pane    │ │ (v1)    │ │ Orchestr.│ │
    오버헤드 마운트)    │  │ +EKF    │ │         │ │         │  │
                        │  └────┬────┘ └────┬────┘ └────┬────┘  │
                        └───────┼───────────┼───────────┼───────┘
                                │      PathML(JSON)     │ TCP/JSON Lines
                     추정 상태  │           ┌───────────┴───────────┐
                     (30–90Hz)  │           ▼                       ▼
                                │  ┌────────────────┐   ┌────────────────────┐
                                │  │ myCobot 280 Pi │   │ AGV                │
                                │  │ (내장 RPi:     │   │ ESP32(ACEBOTT 쉴드: │
                                │  │  ArmBridge.py) │   │  TCP브리지+PWM+워치독)│
                                │  │  펜+스프링홀더 │   │  메카넘 4륜          │
                                │  └────────────────┘   └────────────────────┘
                                │        ▲ 종이 위 드로잉        ▲ 바닥 주행
                                └────────┴───── 카메라가 둘 다 관찰(폐루프) ──┘
```

- **PC(HMI)가 마스터**: 경로 생성·비전 추정·오케스트레이션. 실시간성이 필요한 저수준 제어(모터 PWM)는 AGV 온보드(ESP32)에 위임 — *주파수 영역 역할 분담* 원칙(§8.1).
- 로봇팔/AGV는 **브리지 서버**(Python, RPi)가 표준 프로토콜(§3)로 감싼다. HMI는 하드웨어 종류를 모른다 — 목(mock) 브리지로 교체하면 v1 시뮬레이터처럼 동작한다.

## 2.5 소프트웨어 아키텍처 (PC HMI 내부 — 스레드·데이터 흐름·상태기계)

### 계층과 의존 규칙 (v1 원칙 계승)
```
┌─ App (MFC) ─────────────────────────────────────────────────┐
│  UI 계층: SketchView · VisionPane · HardwarePane · RobotSim │
│  제어 계층: MotionOrchestrator(상태기계) · AccuracyReporter │
│  I/O 계층: CaptureThread(MF) · VisionThread · BridgeClient×2│
├─ Core (정적 lib, 무의존) ───────────────────────────────────┤
│  CanvasModel · PathPlanner · MarkerDetector · KalmanFilter  │
│  FrameTransform · PathTracker · GCodeWriter                 │
└─ protocol/ (DTO 정의 — App와 브리지가 공유하는 유일한 계약) ┘
```
- 의존 방향은 **UI→제어→I/O→Core 단방향**. Core는 여전히 windows.h를 모른다.
- 네트워크·카메라·타이머 등 부작용은 전부 I/O 계층에 격리 — Core 함수는 "값 입력→값 출력"이라 전 모듈 단위테스트 가능.

### 스레드 모델 (총 5종)
| 스레드 | 주기 | 하는 일 | UI와의 통신 |
|---|---|---|---|
| UI (메인) | 이벤트 | 모든 MFC 객체 소유, 렌더링, 오케스트레이터 실행 | — |
| CaptureThread | 90fps | MF ReadSample → MJPEG 디코드(RGB32) | 최신 프레임을 **트리플 버퍼**에 게시 |
| VisionThread | 프레임 도착 시 | MarkerDetector → FrameTransform → **EKF 갱신** | 추정 상태 스냅샷(POD 구조체)을 원자 교체 + PostMessage로 통지 |
| BridgeClient ×2 (Arm/AGV) | 요청/이벤트 | TCP 송수신, 재접속, 하트비트 | 이벤트를 PostMessage로 마샬링 (v1 카메라 패턴 동일) |
| 워치독 타이머 | 100ms | 브리지 연결·프레임 신선도 감시 | 이상 시 오케스트레이터에 FAULT 이벤트 |

원칙: **EKF는 VisionThread에서만 갱신**(단일 작성자), 다른 스레드는 스냅샷만 읽는다 — 락 최소화. AGV로 나가는 `set_pose_estimate`는 VisionThread가 갱신 직후 직접 송신(UI 경유 없음 — 지연 최소 경로).

### MotionOrchestrator 상태기계
```
IDLE ─시작─▶ CHECK(브리지 hello·캘리브 파일) ─▶ ACT1_HOMOGRAPHY(자동)
 ─▶ ACT1_DRAWING(팔 실행, 진행률) ─▶ ACT1_REPORT(오차맵 생성)
 ─▶ TRANSITION(카메라 이동 대기 — 사용자 확인 버튼) ─▶ ACT2_HOMOGRAPHY(자동)
 ─▶ ACT2_MISSION(EKF 수렴 확인 → follow_path) ─▶ DONE
어느 상태에서든: E-STOP ─▶ FAULT(전 브리지 estop 브로드캐스트, 사유 표시, 수동 리셋)
가드 예: ACT2_MISSION 진입은 "EKF NIS 정상 + 마커 검출률 > 90% (직전 3초)"일 때만.
```

### 로깅·리플레이 (장비 SW 관행)
- 세션당 `logs/session_*.jsonl`: 모든 프레임 타임스탬프, 마커 관측, EKF 상태/공분산, 송수신 명령을 기록.
- **리플레이 모드**: 로그를 입력으로 VisionThread 파이프라인을 오프라인 재실행 — 하드웨어 없이 EKF 튜닝(Q 스윕)과 버그 재현 가능. Sprint 3 산출물에 포함.

### 원격(브리지) 측 구조
- **ArmBridge(Python, myCobot RPi)**: 스레드 2개 — TCP 수신(명령 큐 적재) / 실행기(큐 소비, 보간·소프트리밋 검사, pymycobot 호출, progress 이벤트 발행). `stop`은 큐 폐기 + 현재 세그먼트 감속.
- **ESP32 펌웨어(Arduino)**: 태스크 2개 — NetTask(TCP 수신·JSON 파싱·상태 갱신) / ControlTask 50Hz(PathTracker → 메카넘 믹싱 → PWM). 공유 상태는 뮤텍스 1개, 워치독은 ControlTask에서 검사.

## 2.6 드로잉 지그와 마커 배치 (명확화)

- 종이 기준 마커 4개는 **종이가 아니라 지그(클립보드형 드로잉 베드)에 부착**한다. 종이 교체 = 지그에 새 종이를 끼우는 것 — 마커가 움직이지 않으므로 **재캘리브레이션 불필요**.
- 지그는 책상에 고정(양면테이프/클램프). {paper} 원점은 지그 좌상단 마커 기준 — 종이 위치 공차는 지그의 모서리 스토퍼로 ±1mm 이내 관리.
- 드로잉 가능 영역 = 종이 ∩ 로봇 도달범위 ∩ 소프트리밋 마진. **마커 사각형은 좌표 앵커일 뿐 드로잉 경계가 아니다.**
- 마커 규격: 기준·스테이션 40mm, AGV 상판 80mm, 펜 상단 25mm (클로즈업/오버헤드 각 거리에서 최소 20px 확보 기준 — Sprint 3 실측 후 조정).

## 3. 통신 프로토콜 (Bridge Protocol v1 — Sprint 1 확정 대상)

TCP, 한 줄에 JSON 객체 하나(JSON Lines), UTF-8. PC=클라이언트, 브리지=서버(포트: Arm 9101, AGV 9102).

```jsonc
// 공통 요청 골격
{ "id": 17, "cmd": "...", "params": { } }
// 공통 응답
{ "id": 17, "ok": true, "result": { } }        // 실패 시 ok:false, "error": "메시지"
// 브리지 → PC 자발 이벤트 (id 없음)
{ "event": "progress", "data": { "path": 3, "of": 12, "pct": 41.2 } }
```

| cmd | 대상 | params | 설명 |
|---|---|---|---|
| `hello` | 공통 | — | 버전·능력 교환. `{"result":{"device":"mycobot280","proto":1}}` |
| `home` | Arm | — | 홈 포즈 이동 |
| `draw_paths` | Arm | `paths:[[{x,y}..]..], frame:"paper", feed_mm_s` | 종이 좌표(mm) 폴리라인 배열 실행 |
| `stop` | 공통 | — | 즉시 정지(감속), 큐 폐기 |
| `estop` | 공통 | — | 비상정지: 토크 해제/모터 프리 |
| `follow_path` | AGV | `path:[{x,y}..], frame:"floor", v_max` | 바닥 좌표(m) 경로 추종 시작 |
| `set_pose_estimate` | AGV | `x,y,theta,cov[9],t_cam` | PC의 EKF 추정을 AGV에 주입(폐루프) |
| `telemetry` | 공통 | — | 상태 조회 (관절각/엔코더/배터리) |

규칙:
- 모든 좌표는 **SI(mm 또는 m) + 명시된 frame**. 픽셀 좌표는 프로토콜에 절대 등장하지 않는다(§4).
- `set_pose_estimate`의 `t_cam`은 캡처 시각(PC 단조시계). 브리지는 자신의 수신 시각과의 차로 지연을 추정에 반영(§7.5).
- 워치독: AGV는 `follow_path` 중 `set_pose_estimate`가 500ms 끊기면 자동 정지.

## 4. 좌표계 정의 (Frame Tree)

```
{cam}   카메라 픽셀 → (내부 파라미터 K, 왜곡 보정) → 정규화 이미지 좌표
{floor} 바닥 평면 (m)   ← {cam}에서 바닥 기준 마커 4개로 호모그래피 H_cf
{paper} 종이 평면 (mm)  ← {cam}에서 종이 모서리 마커 4개로 호모그래피 H_cp
{robot} myCobot 베이스 (mm) ← {paper}와 4점 대응 → Kabsch(SVD)로 강체 변환 T_pr
{agv}   AGV 차체 (진행방향 +x) ← {floor}에서 상판 SM-Tag 자세로 결정
```

- **캘리브레이션 절차** (B 담당, 도구는 Python/OpenCV 허용):
  1. 카메라 내부 파라미터: 체커보드 20장 → K, 왜곡계수 → `calib/camera.json`
  2. 평면 호모그래피: 기준 마커 4개의 픽셀↔실측 좌표 → DLT → `calib/floor.json`, `calib/paper.json`
  3. 종이↔로봇: 로봇이 4점을 콕콕 찍으면(조그) 그 점을 카메라로 관측 → Kabsch로 T_pr → `calib/robot.json`
- 실행 시 HMI는 JSON 캘리브레이션 파일만 읽는다. **Core::FrameTransform은 호모그래피 적용·Kabsch를 자체 구현**(SVD 포함 — 3×3이라 야코비 회전법으로 충분, 단위테스트 대상).

## 5. Vision 서브시스템 (담당 B)

### 5.1 캡처
- AR0234(SVPRO)는 **USB 2.0 + UVC**. 네이티브 1920×1200, **90fps는 MJPEG에서만 가능** — 대역폭 계산으로 확정: YUY2 1920×1200×2B×90fps ≈ 415MB/s로 USB2(실효 ~40MB/s)를 10배 초과. 비압축은 물리적으로 불가.
- 따라서 캡처 경로는 **MJPEG 고정**: v1 Media Foundation 캡처 확장 + MJPEG Decoder MFT(OS 내장)로 RGB32 변환. IMFSourceReader에 RGB32 출력 타입을 요청하면 MF가 디코더를 자동 삽입한다.
- 글로벌 셔터라 이동 중 마커도 기하 왜곡 없음 → 코너 기반 자세 추정의 전제.

### 5.2 SM-Tag: 자체 정사각 fiducial 마커 (v1 자산 재사용)
- 구조: 검은 테두리 + 내부 4×4 비트(16비트 중 12 데이터 + 4 방향/패리티). 사전 정의 ID 8개면 충분(AGV 1, 바닥 4, 종이 4는 ID 공유 가능, 펜 끝 1).
- 검출 파이프라인(전부 기존 Core 함수 조합):
  `Grayscale → 적응 이진화(신규: 타일별 Otsu) → TraceContours → 폴리곤 근사(RDP) → 4각형 필터 → 호모그래피로 정면화 → 비트 샘플링 → ID/방향 디코드 → 코너 서브픽셀 정련(신규)`
- 출력: 마커 ID, 4코너 픽셀 좌표, (평면 가정 하) {floor}/{paper} 좌표의 위치+방위각.
- 실패 프레임은 그냥 버린다 — 결측은 EKF가 예측으로 메운다(§7).

### 5.3 성능 예산
- 90fps = 11.1ms/frame. 예산: MJPEG 디코드 ≤ 4ms + 검출 ≤ 5ms (1920×1200, 단일 스레드) — ROI 추적(직전 검출 위치 주변만 탐색)으로 달성.
- 미달 시 폴백 순서: ① 검출만 반해상도 ② 60fps로 하향 (EKF는 프레임레이트 변동을 Δt로 자연 흡수하므로 설계 변경 없음).

### 5.4 카메라 1대 · 2구역 문제 (명시적 결정)
- 오버헤드 1대로 AGV 아레나(~2×1.25m, **1.04mm/px**)와 종이(A5급)를 동시에 충분한 해상도로 볼 수 없다.
- **결정: 데모를 2막 구성으로 분리**하고 카메라를 재배치한다.
  - 1막(로봇팔): 카메라를 종이 위 60–80cm → **~0.2mm/px** (정확도 검증 §7.2의 전제)
  - 2막(AGV): 카메라를 아레나 오버헤드 → 1mm/px (EKF 추적에 충분)
  - 재배치 후 각 막 시작 시 해당 평면 호모그래피 재캘리브레이션(기준 마커 4개가 시야에 있으므로 자동, < 1초)
- 확장 옵션(선택): 저가 웹캠 1대를 추가해 동시 2구역 — Sprint 5에서 여유 있을 때만.

## 6. 모션 서브시스템 (담당 C)

### 6.0 엔드이펙터 구성 (Adaptive Gripper 보유 반영)
- **Sprint 2 기본: 펜 홀더 플랜지 직결 마운트.** 그리퍼 파지보다 강성이 높아 드로잉 정확도에 유리하고, 그리퍼 무게만큼 페이로드 여유 확보.
- **Sprint 5 스트레치: Adaptive Gripper로 펜 자동 픽업.** 데모 시작 시 팔이 펜 스탠드에서 스프링 홀더를 집어 드는 연출 — "툴 체인지" 스토리. 단 파지 강성/반복성이 드로잉 오차에 미치는 영향을 클로즈업 검증(§7.2)으로 측정 후 채택 여부 결정. 그리퍼 장착 시 총 엔드이펙터 질량(그리퍼+홀더+펜) < 200g 확인 필수.

### 6.1 myCobot 드로잉 실행기 (ArmBridge.py)
- 입력: `draw_paths` (종이 mm). 변환: T_pr로 {robot} 좌표 → pymycobot `send_coords`(직교) 사용, 세그먼트가 긴 경우 보간 분할(5mm 간격).
- Z 전략: pen-down 높이는 **캘리브레이션이 아니라 스프링이 흡수** — 명령 Z는 종이보다 2–3mm 아래로 '살짝 과다' 지령하고 스프링 순응으로 접촉 유지(§6.2).
- 속도: 드로잉 20mm/s, 이동 60mm/s에서 시작(안전 우선, Sprint 5에서 상향 튜닝).
- 소프트리밋: {robot} 좌표로 작업 영역 AABB를 정의, 벗어나는 명령은 브리지가 거부.

### 6.2 스프링 펜 홀더 — 순응 요소의 실물 적용
- 설계: 펜이 홀더 내에서 축방향으로 ±4mm 슬라이드, 압축 스프링(약 1–2N @ 2mm)이 종이에 눌러줌. 3D 프린트(PETG) 2피스 + 스프링 + 고정나사.
- **무게 제한: 홀더+펜 합계 < 150g** — myCobot 280 페이로드가 250g(공식)이므로 여유 40%를 남긴다. 정밀도가 페이로드에 민감한 저가 암이라 이 마진이 드로잉 품질을 지킨다.
- 이론 연결: 하드(로봇 강체)-소프트(종이/펜) 계면에 **직렬 순응 요소를 삽입**해 Z 오차·표면 굴곡을 기계적으로 흡수 — SEA(Series Elastic Actuator)의 최소 사례. 로봇 Z 반복정밀도(±0.5mm급)와 종이 평탄도 오차가 스프링 작동 범위(±4mm) 안에 들어오므로 폐루프 Z 제어가 불필요해진다. *"제어로 풀 문제를 기계 설계로 제거"* — 설계 리뷰에서 강조할 결정.

### 6.3 AGV 제어 계층 (확정 하드웨어: ESP32 + ACEBOTT 쉴드 + 메카넘 4륜)

```
PC EKF 추정(30–90Hz) ──set_pose_estimate/follow_path──▶ ESP32 (WiFi 내장, AgvBridge 겸 펌웨어)
        (TCP JSON Lines, RPi 불필요 — 계층 하나 제거)      │ 홀로노믹 경로추종 50Hz
                                                           │ (vx, vy, ω) → 메카넘 역기구학
                                                           ▼
                                              ACEBOTT 쉴드: 4모터 PWM (드라이버 내장)
                                              워치독: pose 500ms 결측 → 정지
```

- **메카넘 = 홀로노믹**: 비홀로노믹용 pure pursuit를 폐기하고 **Core::PathTracker(홀로노믹 경로추종)**로 교체.
  - 추종 법칙: 경로상 목표점 대비 위치 오차의 P 제어 + 경로 접선 피드포워드. 헤딩은 위치와 **독립 제어** — 교차로/꺾임에서 **제자리 회전** 가능.
  - 이점: 그린 경로의 예각 꺾임을 회전반경 제약 없이 그대로 재현 — "그린 대로 움직인다"는 데모 정합성이 차동구동보다 우수.
  - 메카넘 역기구학(표준 X 배치):
    `w_fl = (vx − vy − (L+W)·ω)/r`, `w_fr = (vx + vy + (L+W)·ω)/r`,
    `w_rl = (vx + vy − (L+W)·ω)/r`, `w_rr = (vx − vy + (L+W)·ω)/r`
- **엔코더 부재/슬립 전제**: ACEBOTT 쉴드 구동계는 휠 오도메트리를 신뢰할 수 없고, 메카넘은 원래 슬립이 커서 오도메트리 품질이 낮다 → **비전 EKF가 유일한 1차 측위**라는 기존 설계가 이중으로 정당화됨. 개루프 휠 속도 + 비전 폐루프 위치 보정 구조.
- ESP32 펌웨어(Arduino C++): TCP 서버 + JSON 파싱 + 역기구학 + PWM + 워치독. RPi 계층 제거로 지연 스택 단순화(WiFi 1홉).
- **NUCLEO-F103RB의 역할 재배정**: AGV 저수준이 ESP32로 확정되면서 STM32는 v2 필수 경로에서 제외. §13 연구 트랙의 1-DOF 임피던스 테스트베드로 배정 (하드웨어를 이유 없이 끼워 넣지 않는다 — v1 교훈).
- 별도 Raspberry Pi: **예비**(카메라 제2구역 스트리머 또는 CI 러너 후보). v2 필수 경로에 없음.

## 7. 상태추정 설계 (담당 B — 칼만 문서 전면 적용)

### 7.1 AGV EKF (메카넘 반영)
- **상태** x = [x, y, θ, vx, vy, ω]ᵀ (6차원, {floor} 기준 — 홀로노믹이라 횡속도 vy가 독립 상태)
- **프로세스 모델**: 바디 프레임 등속. [ẋ, ẏ] = R(θ)·[vx, vy], θ̇ = ω, (vx, vy, ω)는 랜덤워크. 유니사이클 대비 vy 추가 외에 구조 동일 — 슬립이 큰 메카넘 특성상 속도 상태의 Q를 크게 잡는 것이 출발점.
- **측정 모델**: SM-Tag 검출 → z = [x_m, y_m, θ_m]ᵀ (H는 앞 3상태 선택 + θ 랩어라운드 처리)
- **Q/R 튜닝 절차** (칼만 문서 §2.7 그대로):
  1. R: AGV 정지 상태에서 마커 측정 2분 수집 → 표본 공분산 (θ는 각도 통계 주의)
  2. Q: 로그 스케일 스윕하며 **NIS(정규화 혁신 제곱)가 χ²(3) 95% 구간을 통과**하는 지점 선택
  3. 혁신 백색성 검정을 CI 통합테스트에 포함 — *"필터가 잘 도네"가 아니라 통계로 합격 판정*
- **수치**: 공분산 갱신은 조셉 형식. P 대칭화 매 스텝. (Core 단위테스트: 합성 궤적으로 NEES 검정)
- **결측 처리**: 검출 실패 프레임은 예측만 수행. 연속 결측 > 500ms → AGV 워치독 정지(§3).

### 7.2 로봇팔 정확도 검증 (측정 방법 정정)
- **오버헤드 원거리 뷰(1mm/px)로는 2mm급 오차를 측정할 수 없다** — 측정 도구의 분해능이 측정 대상 오차와 같은 자릿수면 무효. 따라서 팔 검증은 반드시 **종이 클로즈업 배치(§5.4 1막, ~0.2mm/px)**에서 수행한다.
- 방법 2가지 병행:
  1. **실시간**: 펜 홀더 상단 소형 SM-Tag 추적 → {paper} 궤적 vs 명령 궤적(타임스탬프 정합) 오차 맵
  2. **오프라인(더 정밀)**: 완성된 드로잉 종이를 같은 클로즈업 카메라로 촬영 → v1 트레이싱 파이프라인으로 벡터화 → 명령 경로와 형상 비교 (Core 재사용, 소프트웨어 추가 비용 거의 0)
- 산출물: 드로잉 1회마다 `report/accuracy_YYYYMMDD.png` (오차 히트맵 + RMS/최대 오차). **장비 소프트웨어의 검사 리포트 관행**을 그대로 재현하는 기능.

### 7.3 필터 선택 근거
- EKF로 시작(θ 비선형이 약해 1차 선형화로 충분). UKF는 성능 미달 시에만 — 문서의 "중간 비선형이면 UKF 먼저"는 알지만, 측정이 자세 직접 관측이라 비선형성이 약해 EKF가 적정. 이 판단 근거를 코드 주석이 아닌 본 설계서에 남긴다.

### 7.4 시간 동기
- 모든 프레임에 PC 단조시계 타임스탬프. 브리지와는 `hello` 시 왕복지연/오프셋 측정(간이 NTP). 정밀 동기가 필요해지면 Sprint 4에서 재평가.

### 7.5 지연 보상
- 측정 지연(캡처→검출→전송) τ를 측정해 두고, AGV 제어에 쓰는 추정치는 **EKF를 τ만큼 앞으로 예측한 값** 사용 (칼만 문서 §5.3 '지연 보상'의 표준 수법).

## 8. 설계 원칙 (이론 문서 → 실물 결정으로)

### 8.1 주파수 영역 역할 분담 (하이브리드 구동 문서 §5)
| 계층 | 주기 | 담당 | 근거 |
|---|---|---|---|
| 모터 PWM | ~1kHz | ESP32+ACEBOTT 쉴드 | 로컬 하드웨어, 지터 불가 |
| 경로추종 (vx,vy,ω) + 메카넘 믹싱 | 50Hz | ESP32 | WiFi 지터를 워치독+로컬 루프로 흡수 |
| 상태추정 EKF | 30–90Hz | PC | 카메라 프레임레이트 |
| 오케스트레이션/UI | 10Hz | PC HMI | 사람 눈 기준 |
빠른 루프는 아래(로컬)에, 느린 루프는 위(네트워크 너머)에 — 네트워크 지연이 저수준 안정성에 못 닿게 하는 구조. 명령 신호에는 EMA 저역필터+데드존 적용(급격한 지령 변화 억제).

### 8.2 순응 요소 삽입 (동 문서 §5.3)
- 스프링 펜 홀더(§6.2). Z 폐루프 제어 대신 기계적 순응 — 제어 복잡도와 파손 리스크를 동시에 제거.

### 8.3 무의존 Core 원칙 유지 (v1 계승)
- KalmanFilter/MarkerDetector/PurePursuit/FrameTransform은 Core에 순수 C++로, 전부 합성 데이터 단위테스트 동반. 하드웨어·네트워크·OpenCV는 Core에 절대 들어오지 않는다.

### 8.4 실용 예외 (명시적 허용)
- **오프라인 캘리브레이션 도구만 Python+OpenCV 허용**. 이유: 체커보드 내부 파라미터 추정을 자체 구현하는 것은 v2 기간 대비 교육 효익이 낮고, 결과물은 JSON 파일이라 런타임 의존성이 생기지 않는다. — "어디에 시간을 쓸지"의 의도적 결정.

## 9. 안전 설계

1. **E-stop 3계층**: HMI 버튼(`estop` 브로드캐스트) / 브리지 워치독(연결 끊기면 정지) / 물리 전원 스위치(멀티탭).
2. myCobot 소프트리밋(작업 AABB) + 속도 상한을 브리지에서 강제 — HMI 버그가 하드웨어를 해치지 못하게.
3. AGV 주행 구역 물리 경계(매트 밖 = 마커 기준 좌표 밖 → 정지).
4. 데모 중 사람 손 개입 규칙: 정지 후 접근 (myCobot 280은 저출력이지만 습관을 만든다).

## 13. 연구 트랙 (v2와 분리된 스트레치 — research1 반영)

보유 연구물: **"이종 물리 도메인 액추에이터의 단일 포트-해밀토니안 통합과 패시비티 기반 임피던스 제어"** (모터+DEA 단일 Dirac 구조, IDA-PBC, 기호 증명 + 수치검증 스크립트). 저장 위치: `research/ph-hybrid-actuation/`.

**v2 본선에 넣지 않는 이유(명시)**: myCobot은 토크 인터페이스가 없어 임피던스 제어 불가, DEA 하드웨어 부재. 이론을 위해 본선 스코프를 늘리는 것은 v1 실패 패턴.

**정당한 접점 2개**:
1. **NUCLEO-F103RB 1-DOF 테스트베드** — 논문의 "실물 검증 경로" 1단계. DC 모터+엔코더+직렬 스프링(SEA) 1관절을 F103RB로 구동, 에너지 성형 기반 임피던스 제어(논문 모터 서브시스템 PH 모델 그대로)를 실측. DEA 없이도 "PH 모델링→IDA-PBC→실물"의 최소 루프를 닫는다. 필요 부품: DC 기어모터+엔코더, 모터드라이버(L298N급), 스프링, 커플링 — 소액. **단, F103RB의 1순위 배정은 §15(EMG 프론트엔드)이고, 이 테스트베드는 여유 보드 확보 시의 2순위다.**
2. **스프링 펜 홀더(§6.2)의 이론적 서사** — 하드-소프트 계면에 순응 요소를 삽입하는 논문 §1.1 문제 ②·③의 실물 최소 사례로 인용.

시간 규칙: 연구 트랙은 v2 게이트가 그린일 때만 진행(주당 상한 1일). Jira 라벨 `research`.

## 15. STM32 = EMG 프론트엔드 (v2.1 브릿지, 확정 배정)

**결정**: NUCLEO-F103RB의 1순위 용도는 **표면 근전(sEMG) 취득 보드**다. AGV 저수준이 ESP32로 확정되며 STM32가 v2 본선에서 빠졌으나(§6.3), "이유 없이 끼우지 않는다" 원칙에 따라 억지 역할 대신 **다음 프로젝트(EMG 폐루프 데모)의 심장**으로 예약한다. 이는 SketchMotion 인프라를 재사용해 개발자의 진로 서사(생체신호→실시간 폐루프 디코딩)로 잇는 다리다.

### 왜 이게 타당한가 (억지가 아닌 이유)
- STM32의 교과서적 용도 = **하드 리얼타임 ADC + 타이머 + DSP**. sEMG 1kHz 샘플링이 정확히 그 일이다.
- SketchMotion의 브리지 아키텍처에 **`EmgBridge`를 새 입력원으로 추가**하면 끝 — Arm/AGV 브리지와 동일 패턴(§2.5, §3). 폐루프 오케스트레이션·로깅·MockBridge 하네스가 그대로 재사용된다.
- 신호 처리 본체는 **PC의 `Core::EmgProcessor`**(무의존 Core 원칙 유지). STM32는 샘플링·전송만.

### 파이프라인
```
MyoWare 2.0 sEMG ─(ENV/RAW)→ STM32 F103RB (ADC 1kHz, PA0)
   → USART2 VCP(115200) JSON Lines {"event":"emg","data":{"raw":0..4095,"t":ms}}
   → emg_bridge(PC, 시리얼↔TCP 어댑터) → 9103 TCP
   → App BridgeClient → Core::EmgProcessor
        (고역 DC제거 → 정류 → 저역 엔벌로프 → 1D 칼만 → MVC정규화 → 데드존 → 비례/히스테리시스 제스처)
   → 그리퍼/커서/로봇 비례 제어  (Arm/AGV estop이 최우선)
```

### 구현 상태 (이번 스프린트)
- ✅ `Core::EmgProcessor` (+ 단위테스트 12종: 비례성·데드존·히스테리시스·MVC·리셋) — **556 checks 통과**
- ✅ 프로토콜 §3 EMG(9103) 확장: `start_stream`/`stop_stream`/`calibrate_mvc`, `emg`/`mvc` 이벤트
- ✅ `firmware/emg_stm32/` 펌웨어 스켈레톤 + 배선/CubeMX 설정 README
- ✅ `MockBridge.exe emg` — 합성 근수축 스트림, 하드웨어 없이 파이프라인 검증
- ✅ `tools/emg_e2e` — MockBridge emg → EmgProcessor 가상 통합 **PASS** (이완<수축 비례출력, 제스처 검출)
- ⏳ 실물: MyoWare 2.0 구매 + STM32CubeIDE 프로젝트화 + `emg_bridge` 시리얼 어댑터 (하드웨어 확보 후)

### 이력서 한 줄 (BCI 아카이브 08번)
"표면 근전(sEMG)을 실시간 디코딩해 로봇을 비례 제어하는 폐루프 — 전처리 + 상태추정(칼만) + 제어(P/데드존/히스테리시스). SketchMotion 브리지 인프라 재사용, 하드웨어 없이 목 브리지로 파이프라인 검증."
