# SketchMotion v2.0 계획서 — 4인 팀 · 실물 하드웨어 통합

> 작성 2026-07-06 · v1.0(시뮬레이션 완결) 기반 · Jira Epic: KAN-16 (v2)
> 설계 상세는 [DESIGN_V2.md](DESIGN_V2.md) 참조.

## 1. 목표 (한 문장)

**"캔버스에 그린 것을, 카메라가 지켜보는 가운데, 실물 로봇팔이 종이에 그리고 실물 AGV가 바닥에 주행한다."**

v1.0의 파이프라인(드로잉→비전→모션)은 그대로 두고, 출력단을 시뮬레이션에서 실물로 교체한다.
핵심 기술 격상: **개루프 → 비전 폐루프(vision-in-the-loop)**. 카메라가 로봇/AGV의 실제 상태를 추정(칼만 필터)하고, 그 추정으로 제어를 보정한다. 이것이 v1과 v2를 가르는 이력서 문장이다:
- v1: "그림을 G-code로 바꿨다"
- v2: **"90fps 글로벌 셔터 카메라로 로봇의 실제 위치를 EKF로 추정하고, 명령 대비 오차를 실시간 보정·리포트하는 폐루프를 만들었다"**

## 2. 보유 장비와 역할

| 장비 | 역할 | 비고 |
|---|---|---|

| myCobot 280 Pi (+ **Adaptive Gripper**) | **실물 드로잉 로봇팔** (6-DOF) | 내장 RPi에서 브리지 서버. 기본: 펜 홀더 플랜지 직결, 그리퍼는 펜 자동 픽업 스트레치(설계 §6.0). 펌웨어 버전 Sprint 1 확인 |
| AGV: **ESP32 + ACEBOTT 스마트카 쉴드 + 메카넘 4륜** | **바닥 경로 주행** — 홀로노믹(제자리 회전) | ESP32 WiFi 내장 → 브리지 겸 펌웨어, RPi 계층 불필요. 상판 SM-Tag (설계 §6.3) |

| Raspberry Pi (별도) | 예비 (2구역 동시 카메라 스트리머/CI 러너 후보) | v2 필수 경로 아님 |

배정 근거: 재니는 이 프로젝트에서 가장 수학 밀도가 높은 모듈(EKF·마커·캘리브레이션)을 맡는 것이 팀 전체 리스크를 최소화한다(관련 학습 이력·장비 이해도 보유). PO 겸직은 Jira 운영을 이미 하고 있기 때문. 팀원 영입 시 각 역할의 "필요 스킬" 열을 모집 기준으로 사용.

운영 규칙 (v1에서 확립한 것 유지 + 팀 규칙 추가):
- Jira KAN 보드 = 단일 소스 오브 트루스. 스프린트 2주. 커밋에 `KAN-XX:` 필수.
- **인터페이스 우선**: 사람 경계 = 모듈 경계. 스프린트 1에서 프로토콜/좌표계 스펙을 확정하고, 이후 변경은 Confluence 결정 기록(ADR) + 전원 합의로만.
- PR 리뷰 1인 이상 필수. `main`은 항상 빌드+테스트 그린. 브랜치명 `feature/KAN-XX-설명`.
- 매 스프린트 말 **통합 리허설**(전 모듈 연결 데모) — 마지막에 몰아서 통합하다 터지는 것 방지.

## 8. v1 대비 아키텍처 원칙 유지 사항

- Core(무의존 C++)에는 계속 알고리즘만: **KalmanFilter, MarkerDetector, PurePursuit가 Core에 추가**되고 단위테스트로 검증된다. 하드웨어 I/O는 전부 App/브리지 계층.
- 시뮬레이터는 삭제하지 않는다 — **하드웨어 없는 팀원도 개발 가능한 목(mock) 실행기**로 승격 (실물과 같은 프로토콜을 말하는 가짜 브리지).
- 문서: 설계서(docs/) = 단일 원본, Confluence = 공유 요약, Jira = 상태.
